Как обновлять сайт в будущем
После пуша изменений в GitHub:
cd /var/www/onestep && git pull && npm ci && npm run build && pm2 reload onestep

Как обновлять сайт
После пуша изменений в GitHub — просто:

ssh root@xxx.21.60.3
cd /var/www/onestep && ./deploy.sh
Скрипт сам всё сделает по порядку. Если в git нет новых коммитов — он сделает бэкап и выйдет, ничего не ломая. Если сборка упадёт — автоматически откатит код и БД на рабочую версию. Весь ход пишется в /var/www/onestep/deploy.log.

⚠️ Важно про твой рабочий процесс с БД: изменения схемы должны приезжать готовыми миграциями в git. То есть локально: правишь schema.prisma → npm run migration:create → применяешь локально → коммитишь папку миграции → пушишь. На сервере deploy.sh их накатит через migrate deploy. Не запускай на сервере prisma migrate dev — он для разработки и может пересоздать базу.

Как восстановить БД из бэкапа
Посмотреть список бэкапов:


cd /var/www/onestep && ./restore.sh
Восстановить самый свежий:


./restore.sh latest
Восстановить конкретный (имя из списка):


./restore.sh dev.db.20260707-150914
Скрипт перед заменой сам сохранит текущую базу в backups/dev.db.before-restore-<время> — так что даже ошибочное восстановление можно отменить, накатив этот файл обратно тем же ./restore.sh.

Детали
Бэкапы: /var/www/onestep/backups/, имя dev.db.<дата>-<время>. Хранятся последние 10, старые удаляются автоматически.
Ручной бэкап в любой момент без деплоя:

sqlite3 /var/www/onestep/prisma/dev.db ".backup '/var/www/onestep/backups/dev.db.manual-$(date +%Y%m%d-%H%M%S)'"
Скрипты лежат рядом с проектом и не в git (untracked) — обновлениям через git pull не мешают.
